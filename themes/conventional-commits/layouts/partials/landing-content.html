Conventional Repositories 1.0.0

Summary

The Conventional Repositories specification is a lightweight convention on top of repository names. It provides an easy set of rules for creating an explicit and discoverable naming structure; which makes it easier to write automated tools on top of. This convention dovetails with organization-wide naming strategies, by describing the type, project, and optional component of each repository.

The repository name should be structured as follows:

[<type>-]<project>[-<component>]


The repository name may contain the following structural elements, to communicate intent to the consumers of your codebase:

  1. type: an OPTIONAL noun that classifies the repository. When present, it SHOULD be one of the recognized types (e.g., app, lib, docs).
  2. project: the name of the project or domain the repository belongs to, providing context and grouping. This is the only REQUIRED segment.
  3. component: an OPTIONAL segment that further describes the repository's role within a project, such as a specific microservice, module, or deployment target.
  4. types other than those listed below are allowed, for example config, docs, infra, or tool based on your organization's needs.
  5. Additional segments beyond type-project-component MAY be appended when necessary, though a flat naming structure is RECOMMENDED.

Types are not mandated by the Conventional Repositories specification. A repository name without a type prefix is simply a project name — this allows both typed and untyped names to coexist in the same organization. Well-known projects such as nextjs, prisma, or shadcn-ui are valid names without a type prefix.

Repository Name Format

The format ensures consistency by optionally prefixing a type, followed by a project name, and an optional component. Each segment is separated by a single hyphen.

[<type>-]<project>[-<component>]

Where:

  • <type> — An OPTIONAL noun that classifies the repository's purpose. When present, examples include: app, service, api, lib, cli, docs, infra, plugin, template, example.
  • <project> — The name of the project, system, or domain the repository belongs to. MUST be present.
  • <component> — An optional qualifier that narrows the scope within a project (e.g., frontend, backend, worker).

Hyphens within the project or component name are allowed, but consecutive hyphens (e.g., app--name) MUST NOT be used. When a type prefix is present, the first segment of the name is interpreted as the type; otherwise the entire name is the project name.

Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

  1. A repository name SHOULD consist of at least a project name. A type prefix MAY be prepended to provide additional context.
  2. When a type prefix is present, it MUST be a lowercase noun (e.g., app, lib, docs), followed by a hyphen, followed by the project name.
  3. When no type prefix is present, the entire repository name is the project name. Tools SHOULD NOT assume a type prefix is required.
  4. The type prefix, when present, MUST be lowercase. Uppercase or mixed-case types are NOT RECOMMENDED.
  5. The type app SHOULD be used when a repository contains an end-user application deployable as a standalone unit.
  6. The type service SHOULD be used when a repository contains a backend service or microservice.
  7. The type api SHOULD be used when a repository defines or implements an API surface.
  8. The type lib SHOULD be used when a repository contains a shared library intended for reuse across projects.
  9. The type cli SHOULD be used when a repository provides a command-line interface tool.
  10. The type docs SHOULD be used when a repository is dedicated to documentation.
  11. The type infra SHOULD be used when a repository contains infrastructure-as-code, deployment configurations, or environment definitions.
  12. The type plugin SHOULD be used when a repository extends the functionality of another system.
  13. The type template SHOULD be used when a repository serves as a starter or scaffold for new projects.
  14. The type example SHOULD be used when a repository contains sample code or sandbox projects.
  15. Types other than those listed above MAY be used. Organizations SHOULD define and publish their own type catalog.
  16. When a type prefix is present, the project name MUST follow the type, separated by a single hyphen. The project name SHOULD be concise and descriptive.
  17. An OPTIONAL component segment MAY appear after the project name, separated by a hyphen. The component further qualifies the repository's role (e.g., app-billing-frontend).
  18. Hyphens within the project or component names are permitted, but the double hyphen -- MUST NOT appear anywhere in the name.
  19. The repository name MUST NOT contain characters other than lowercase letters (a-z), digits (0-9), and hyphens (-).
  20. The project and component segments MUST each contain at least one character.
  21. Repository names SHOULD NOT exceed 100 characters in total length.
  22. Tools that consume repository names SHOULD treat the type prefix as case-insensitive for matching purposes; however, the canonical form MUST be lowercase.
  23. When migrating existing repositories to this convention, organizations SHOULD add aliases or redirects from the old names to the new names to preserve links and references.
  24. Parsing ambiguity: when a repository name could be interpreted as having a type prefix (e.g., ui-shadcn) but the intent is a project name, the first segment MAY be treated as part of the project name. Organizations SHOULD document whether they require type prefixes to avoid ambiguity.

Why Use Conventional Repositories

  • Automatically discovering repositories by type (e.g., finding all lib-* or service-* repositories).
  • Enforcing organization-wide naming standards without additional tooling configuration.
  • Communicating the purpose of a repository at a glance from its name alone.
  • Triggering automated provisioning, CI/CD pipelines, and deployment rules based on naming patterns.
  • Making it easier for people to find the repositories they need across large organizations and multi-project ecosystems.
  • Reducing onboarding friction by providing a predictable repository structure.
  • Enabling search, filtering, and grouping via simple glob patterns (e.g., docs-*, *-billing-*).

FAQ

How should I name repositories during the initial development phase?

We recommend that you choose a project name early and keep it. While renaming is possible, it creates churn for contributors and tooling. Adding a type prefix later is straightforward and does not break existing tooling that supports optional types.

When should I use a type prefix?

Use a type prefix when the repository's role is well-defined and you want to enable type-based discovery and automation. For libraries, frameworks, or standalone tools with a distinct brand (e.g., nextjs, prisma), using only the project name without a type is RECOMMENDED.

What if my project name contains a hyphen?

Hyphens within the project name are permitted. For example, app-user-auth has type app and project user-auth. Without a type prefix, user-auth or shadcn-ui are valid project-only names. The type prefix, when present, provides the unambiguous boundary.

How do I handle monorepos?

For monorepos that contain multiple projects, use a single repository with a descriptive name such as app-monorepo, infra-monorepo, or simply monorepo without a type prefix. Alternatively, use the component segment to denote the subsystem: app-billing-monolith.

Should I rename existing repositories to follow this convention?

Renaming is OPTIONAL but RECOMMENDED for active repositories. Since the type prefix is optional, existing project-only names (e.g., my-tool) are already valid. You can add a type prefix later without breaking compatibility. Configure a redirect or alias from the old name to the new name to avoid breaking existing references.

How does this relate to repository topics or tags?

Repository topics (e.g., GitHub topics) provide metadata orthogonal to the naming convention. Topics can supplement the name with additional classifiers such as language, framework, or team ownership. The naming convention focuses on the structural role of the repository.

What types should I use for internal tooling?

Use cli for command-line tools, app for internal dashboards, lib for shared utilities, and infra for infrastructure configuration. For anything that does not fit, define a custom type and document it in your organization's type catalog, or omit the type prefix entirely.

How do I name documentation-only repositories?

Use the docs type prefix, or simply name the repository after the documentation project (e.g., handbook, api-reference). Examples: docs-handbook, docs-api-reference.

What if two repositories share the same project name?

If two repositories have the same project name, use a type prefix or a component segment to disambiguate. For example, app-billing and lib-billing. If they serve identical purposes, consider consolidating them into a single repository.

Does this convention support versioned repositories?

Version numbers SHOULD NOT appear in the repository name. Repository names represent a living project; use tags, releases, or branches to track versions. If you must include a version (e.g., api-payments-v2), place it at the end as a component segment.

How do I name fork or mirror repositories?

Forks and mirrors SHOULD retain the original repository name to preserve familiarity. If modification is necessary, append a suffix such as -fork or -mirror as the component segment: lib-config-fork.

Can I use this convention with private and public repositories together?

Yes. The convention applies equally to private and public repositories. It is RECOMMENDED to apply the same naming rules across both scopes for consistency.

How do I handle archived repositories?

Archived repositories MAY retain their existing names. If you wish to indicate archival status, append -archive or -legacy as the component segment: app-dashboard-archive. This is OPTIONAL and implementation-defined.

How do tools know whether a prefix is a type or a project name?

Tools MAY use a known type catalog to identify recognized type prefixes. When a type prefix is not recognized, the tool SHOULD treat it as part of the project name. Organizations using type prefixes SHOULD publish their type catalog so tooling can reliably distinguish types from project names in hyphenated repositories.
